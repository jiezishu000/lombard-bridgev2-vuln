How a Single Line of Solidity Can Permanently Lock Cross-Chain LBTC

A CEI Violation in Lombard Finance's BridgeV2

TL;DR

Lombard Finance's BridgeV2.handlePayload() sets a "spent" flag before executing the withdrawal. If the withdrawal fails for any reason, the user's LBTC is permanently stuck with no way to retry.

Root Cause: CEI (Checks-Effects-Interactions) pattern violation — one line of code in the wrong order.

Immunefi V2.2 Classification
Medium — Permanent Fund Freezing

The Vulnerability

The bug is in BridgeV2.handlePayload():

function handlePayload(Payload calldata payload) external {
    // ... validation ...

    // LINE ~593: Mark as spent BEFORE withdrawal
    $.payloadSpent[payload.id] = true;

    // ... later ...
    _withdraw($, chainToAddress, payload);  // This can fail!
}

The payloadSpent[payload.id] = true assignment acts as a replay-protection flag — once set, the same payload cannot be processed again. This is standard for cross-chain bridges.

The problem: It's set before the withdrawal executes, not after.

Impact: Permanent Fund Lock

If _withdraw() reverts for any reason:

  1. Out of gas: Cross-chain calls often have complex execution paths. If the destination chain runs out of gas mid-execution, the withdrawal fails but payloadSpent is already true.
  2. Recipient contract rejects: If the destination address is a smart contract whose onERC721Received() (or similar hook) reverts, the withdrawal fails — but the payload is marked spent.
  3. Bridge paused: If the bridge is paused between the payloadSpent assignment and the _withdraw call.
  4. Any unexpected revert: Any error in _withdraw means the funds are stuck.

✅ payloadSpent[payload.id] = true (cannot retry)
❌ _withdraw() never completed (funds not delivered)
🔒 Result: LBTC permanently locked in the bridge contract

The Fix

Move the state change after the external interaction:

// First, execute the withdrawal
_withdraw($, chainToAddress, payload);

// Then, mark as spent (Effects after Interactions)
$.payloadSpent[payload.id] = true;

This follows the standard CEI (Checks-Effects-Interactions) pattern that every Solidity developer should know. If _withdraw fails, the transaction reverts entirely, and payloadSpent is never set — the user can retry.
